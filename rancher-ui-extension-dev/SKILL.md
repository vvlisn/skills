# Rancher UI Extension Development

This skill helps develop Rancher Dashboard UI extensions. Extensions are packaged Vue libraries that extend Rancher UI with custom functionality via the Extensions API.

## Scope

**✅ Use for**: scaffolding new Rancher extensions; adding products / pages / routes / tabs / actions / panels / cards / table columns; extension folder conventions (`detail/`, `edit/`, `list/`, `models/`, `l10n/`); version compatibility via `rancher.annotations`; building, developer-loading, publishing; top-level vs cluster-level routing.

**❌ Not for**: generic Vue/TypeScript work, core dashboard development in `shell/`, backend / controller / CRD schema work with no Rancher UI scope.

## First moves

1. **Classify the request**: scaffolding, API usage, routing, publishing, migration, or debugging.
2. **Read the relevant doc** from the reference map below before writing code.
3. **Default coding rules** (apply unless the user says otherwise):
   - TypeScript everywhere; `<script setup lang="ts">` (Composition API) by default.
   - Use Options API only when a required mixin forces it (see the mixin list below).
   - Never use raw HTML `<input>` / `<select>` / `<textarea>` / `<table>` — always use Rancher form/table components.
   - Never implement custom save buttons or direct API calls on edit pages — go through `CruResource` + `CreateEditView`.

## Key concepts

### Development app vs extension package

Before any code, the mental model:

| Concept | What it is | Where |
|---|---|---|
| **Development app** | A full Rancher UI shell used to host and hot-reload extensions during development (wraps `@rancher/shell` via npm). | A folder generated by `npm init @rancher/extension@latest <name>`. |
| **Extension / package** | The actual extension source code (many per dev app allowed). | `./pkg/<extension-name>/` inside the development app. |

All extensions inside one development app must share the same `@rancher/extension` tag; mixing tags breaks the build.

### Two product types

| Type | `inStore` | Route pattern | Route name prefix | Scope |
|------|-----------|---------------|-------------------|-------|
| Top-level | `'management'` | `/<PRODUCT>/c/:cluster/…` | `${PRODUCT}-c-cluster-*` | Global, like Fleet |
| Cluster-level | `'cluster'` | `/c/:cluster/<PRODUCT>/…` | `c-cluster-${PRODUCT}-*` | Single cluster, within Cluster Explorer |

Both types require an explicit `plugin.addRoutes(...)` — shell does not auto-generate
resource routes from `configureType`. See [references/products.md](references/products.md#cluster-level-product-complete-example)
for the full cluster-level example and [official docs](https://extensions.rancher.io/extensions/next/usecases/cluster-level-product).

### Edit & detail page pattern

**Edit pages** must use `CruResource` + `CreateEditView` mixin. **Never** implement custom save buttons, raw `<input>` elements, or direct API calls.

| Role | Component / Mixin | Import |
|------|-------------------|--------|
| Form wrapper | `CruResource` | `@shell/components/CruResource.vue` |
| CRUD behavior | `CreateEditView` | `@shell/mixins/create-edit-view` |
| Name/NS/Desc | `NameNsDescription` | `@shell/components/form/NameNsDescription` |
| Text input | `LabeledInput` | `@components/Form/LabeledInput/LabeledInput.vue` |
| Tabs | `Tabbed` + `Tab` | `@shell/components/Tabbed` |

Other standard form pieces (`LabeledSelect`, `KeyValue`, `ArrayList`, `Labels`, `ResourceTabs`) — see [references/edit-detail-patterns.md](references/edit-detail-patterns.md).

Save flow: `CruResource @finish` → `CreateEditView.save()` → `SteveModel.save()` → API. Customize via `registerBeforeHook()` / `registerAfterHook()` or override `actuallySave(url)`.

**Read [references/edit-detail-patterns.md](references/edit-detail-patterns.md) before creating any edit or detail page.**

### Mixins that force Options API

These mixins rely on `this` + `data() / computed / methods`; when you use them, write the component with Options API:

| Mixin | Import | Typical use |
|-------|--------|-------------|
| `CreateEditView` | `@shell/mixins/create-edit-view` | Edit/create pages |
| `ResourceFetch` | `@shell/mixins/resource-fetch` | List pages that batch-load resources |
| `ChildHook` | `@shell/mixins/child-hook` | Before/after-save hooks in nested forms |

Everything else (detail pages, tabs, panels, cards, dashboards, custom pages) should be Composition API + `<script setup lang="ts">`.

### Extension entry pattern

Every extension needs three files wired together:

1. **`index.ts`** — calls `importTypes(plugin)`, sets `plugin.metadata`, registers products via `plugin.addProduct()` and routes via `plugin.addRoutes()`.
2. **product file** — exports `init()` function, uses `$extension.DSL(store, name)` to call `product()` for side-menu registration. Conventionally named `product.ts`, but the file name is arbitrary — e.g. [Kubewarden](https://github.com/rancher/kubewarden-ui/blob/main/pkg/kubewarden/config/kubewarden.ts) uses `./config/kubewarden.ts` and passes it in via `require('./config/kubewarden')`.
   > **Exception — parasitic pattern**: When hooking into an *existing* product (e.g. explorer) rather than creating your own, use `import { DSL } from '@shell/store/type-map'` instead of `$extension.DSL`. See [products.md § Hook-into-explorer](references/products.md#hook-into-explorer-pattern-parasitic-mode) for rationale.
3. **`routing/extension-routing.ts`** — defines Vue Router routes; each route's `meta` should carry at least `product` (see route meta table below).

Use `require('./product')`, not `import product from './product'`. The shell runtime iterates registered products and calls `impl.init(plugin, store)` — it looks up the `.init` **property** on the module object at load time. A default-exported function would type-check fine but silently do nothing (no `.init` property means the shell skips it). The canonical entry file therefore reads:

```ts
import { importTypes } from '@rancher/auto-import';
import { IPlugin } from '@shell/core/types';
import extensionRouting from './routing/extension-routing';

export default function (plugin: IPlugin): void {
  importTypes(plugin);
  plugin.metadata = require('./package.json');
  plugin.addProduct(require('./product'));
  plugin.addRoutes(extensionRouting);
}
```

> **`require(...)` vs `import` — when to use which**
> - `plugin.addProduct(require('./product'))` — **must use `require`**. The shell runtime looks up the `.init` **property** on the module object. An ESM `import product from './product'` would bind to the default export instead of the named `init`, so the product loads silently as a no-op.
> - `plugin.addRoutes(routes)` — **prefer ESM `import`**. Routes are a plain array; both styles are functionally equivalent, but `import` is idiomatic and preserves TypeScript inference. Older examples that use `require('./routing/extension-routing').default` still work.
> - `plugin.metadata = require('./package.json')` — **use `require`**. JSON imports vary across bundler configs; `require` is the most portable form.

And the product file uses a **named** `init` export (never `export default`):

```ts
export function init($extension: IPlugin, store: any): void { /* ... */ }
```

*Parameter naming note:* older official samples (shell 3.0.8 era, e.g. [top-level-product](https://github.com/rancher/ui-plugin-examples/blob/main/pkg/top-level-product/product.ts)) declare the first parameter as `$plugin: IPlugin`; newer production extensions (shell 3.0.11+, e.g. Kubewarden) use `$extension: any`. The two are equivalent — it's a formal-parameter name, not an API distinction.

**Route `meta` fields**:

| Field | Required? | Value | Why |
|---|---|---|---|
| `product` | **Required** | your product name | Tells Rancher which product owns this route (side-menu highlighting, layout). |
| `cluster` | Recommended | `'_'` for top-level, `:cluster` param for cluster-level | Determines whether the cluster context is loaded. Kubewarden's routes omit it and still work. |
| `pkg` | Optional | your extension package name | Enables precise per-extension bundle lazy-loading. See note below. |

> **`meta.pkg` note.** Most official examples ([top-level-product](https://github.com/rancher/ui-plugin-examples/tree/main/pkg/top-level-product), `clock`, `shell-api-demo`, `homepage`) omit `pkg` and still render. Add it only when a complex routing setup hits a shell 404 or when the bundle must lazy-load precisely (e.g. multi-extension dev apps like [primetranslationsimages](https://github.com/rancher/ui-plugin-examples/tree/main/pkg/primetranslationsimages)).

### Extensions API methods

All UI enhancement methods follow the pattern: `plugin.<method>(where, when, options)`.
**All location enums are imported from `@shell/core/types` — never pass raw string literals.**

| Method | `where` locations |
|--------|------------------|
| `addAction` | `ActionLocation.HEADER`, `ActionLocation.TABLE` |
| `addTab` | `TabLocation.RESOURCE_DETAIL_PAGE`, `RESOURCE_EDIT_PAGE`, `RESOURCE_CREATE_PAGE`, `RESOURCE_SHOW_CONFIGURATION`, `CLUSTER_CREATE_RKE2`, `OTHER` |
| `addPanel` | `PanelLocation.DETAILS_MASTHEAD`, `DETAIL_TOP`, `RESOURCE_LIST`, `ABOUT_TOP` |
| `addCard` | `CardLocation.CLUSTER_DASHBOARD_CARD` |
| `addTableColumn` | `TableColumnLocation.RESOURCE` |
| `addTableHook` | `TableLocation.RESOURCE` |

#### Minimal examples (inline — no need to open reference docs for simple cases)

**`addAction` — Header button**

```ts
import { ActionLocation } from '@shell/core/types';

plugin.addAction(
  ActionLocation.HEADER,
  { product: ['explorer'] },
  {
    labelKey: 'myExt.header.action',
    icon:     'icon-pipeline',
    invoke(_opts, _resources) { /* ... */ },
  },
);
```

**`addTab` — Extra tab on a resource detail page**

```ts
import { TabLocation } from '@shell/core/types';

plugin.addTab(
  TabLocation.RESOURCE_DETAIL_PAGE,
  { product: ['explorer'], resource: ['apps.deployment'] },
  {
    name:      'restart-stats',
    labelKey:  'myExt.tab.restartStats',
    weight:    -1,
    component: () => import('./tabs/RestartStats.vue'),
  },
);
```

**`addPanel` — Panel above a resource list**

```ts
import { PanelLocation } from '@shell/core/types';

plugin.addPanel(
  PanelLocation.RESOURCE_LIST,
  { resource: ['apps.deployment'] },
  { component: () => import('./panels/DeploymentBanner.vue') },
);
```

For all option keys, deprecated locations, and per-location variants, open the corresponding reference doc.

#### Additional plugin methods (no `where`/`when` pattern)

| Method | Purpose |
|--------|---------|
| `setHomePage(component)` | Set custom home page component |
| `addUninstallHook(hook)` | Register cleanup on plugin uninstall |
| `addStore(name, register, unregister)` | Register generic Vuex store |
| `addDashboardStore(name, specifics, config, init?)` | Register dashboard-aware Vuex store (used by production extensions like [Kubewarden](https://github.com/rancher/kubewarden-ui) to inject their own Vuex module alongside Rancher's core store) |
| `enableServerSidePagination(config)` | Enable SSP for specified resources |
| `addModelExtension(type, clz)` | *(experimental)* Extend resource models |
| `addNavHooks(hooks)` | Register enter/leave/login/logout hooks |

### LocationConfig (`when` parameter)

A **LocationConfig** object (also accepts a plain string as shorthand for `{ resource: [string] }`). Common keys:

| Key | Type | Example |
|-----|------|---------|
| `product` | Array | `['explorer']` |
| `resource` | Array | `['apps.deployment']` or `['*']` |
| `cluster` | Array | `['local']` |
| `mode` | Array | `['edit', 'create', 'detail', 'config', 'list']` |
| `queryParam` | Object | `{ type: 'digitalocean' }` |

Additional keys — `namespace`, `id`, `hash`, `path`, `context` — see [references/location-config.md](references/location-config.md).

`mode` values also exist as constants in `@shell/config/query-params` (`_EDIT`, `_CREATE`, `_VIEW`, `_CONFIG`, `_LIST`). Prefer the constants over string literals in application code.

### Folder conventions

| Folder | Convention |
|--------|-----------|
| `detail/`, `edit/`, `list/` | File name = Kubernetes resource name |
| `models/` | Extend `SteveModel` from `@shell/plugins/steve/steve-class` |
| `pages/` | Custom (non-resource) Vue pages, imported explicitly in routing files. |
| `l10n/` | YAML translation files (e.g. `en-us.yaml`). Keys merge with core translations — always namespace under your extension prefix. Consume in components via `useI18n(store)` from `@shell/composables/useI18n` (Composition API) or `this.t(key)` on Options API components. |
| `promptRemove/` | File name = resource name |

### Version compatibility

Set in `./pkg/<name>/package.json` under `rancher.annotations`:

| Annotation | Purpose |
|------------|---------|
| `catalog.cattle.io/rancher-version` | Rancher Manager semver range |
| `catalog.cattle.io/kube-version` | Kubernetes semver range |
| `catalog.cattle.io/ui-extensions-version` | Extensions API semver range |
| `catalog.cattle.io/display-name` | Display name on Extensions page |

### Performance & scale (Server-Side Pagination)

Rancher regularly runs against clusters with 10k+ pods / 20k+ nodes. Since **v2.12.0** the Dashboard switched to **Server-Side Pagination (SSP)** for Steve API access — extensions that still `findAll` on large types will hang the UI. **Extensions MUST support both SSP-on and SSP-off** (user-controlled via the `ui-sql-cache` feature flag).

**Scope**: only the Steve API (`/v1/*`, `/k8s/clusters/<cid>/v1/*`). Norman (`/v3/*`) and native Kube (`/apis/*`) requests are unchanged.

**Probe first, branch always**:

```ts
const paged = store.getters['cluster/paginationEnabled'](POD);
// or store.getters['management/paginationEnabled'](MANAGEMENT.CLUSTER)
```

**Three refactor axes**:

| Old | New | When |
|---|---|---|
| `findAll` | **`findPage`** | Type has one global cache; you only need a filtered slice. |
| `findAll` + manual filter | **`PaginationWrapper`** | Same type needs multiple concurrent filtered result sets (cache would collide). |
| `findMatching` / local `matching()` | **`findLabelSelector`** + `shell/utils/selector-typed.ts` `matching()` | Any labelSelector-based filtering. |

**Component swap**: `LabeledSelect` → **`ResourceLabeledSelect`** (with `paginatedResourceSettings`) wherever the user picks a resource.

**Validate both flag states** before shipping. See [references/performance.md](references/performance.md) for the full checklist, example commits from `rancher/dashboard`, and symptom → cause mapping.

### Testing

The official scaffold ships **no test wiring**. Extensions have to opt in — production repos (notably [kubewarden-ui](https://github.com/rancher/kubewarden-ui)) converge on:

- **Unit**: Jest + `@vue/vue3-jest` + `ts-jest`, run from the extension root.
- **E2E**: either **`@rancher/cypress`** (official, page-objects included, `cypress@11.1.0` pinned) or **Playwright in an isolated `tests/` sub-workspace** (own `package.json` + `yarn.lock`).

The single most important Jest config line is the `@shell/*` mapper — without it every shell import throws `Cannot find module`:

```ts
moduleNameMapper: {
  '@shell/(.*)': '<rootDir>/node_modules/@rancher/shell/$1',
  // ...mirror every alias declared in vue.config.js / tsconfig.paths.json
}
```

Extensions must also **test both `ui-sql-cache` states** (SSP on/off) — either by parameterising `cluster/paginationEnabled` getters in specs or by running the E2E suite twice in CI.

See [references/testing.md](references/testing.md) for the full `jest.config.ts` template, the `@rancher/cypress` vs Playwright comparison, and the kubewarden `tests/` directory layout.

## Initializing a new extension

Instead of running the scaffold manually and then patching versions, use the bundled initializer:

```bash
skills/rancher-ui-extension-dev/scripts/init-extension.sh \
  --name <pkg-name> \
  [--shell-version <x.y.z>] \
  [--display-name "Human Readable"] \
  [--minimal] [--yes] \
  [--no-tests] [--with-e2e cypress|playwright]
```

**Defaults**

| Flag | Default | Meaning |
|---|---|---|
| `--shell-version` | `3.0.8` | Verified for Rancher **2.13.x** (incl. 2.13.3). |
| `--rancher-version` | derived from `--shell-version` | See matrix below. |
| `--display-name` | same as `--name` | — |
| `--org` | `<org>` | Injected into `package.json.icon` GitHub URL. |
| `--no-tests` | (off — Jest **installed by default**) | Skip Jest wiring entirely. See [references/testing.md](references/testing.md). |
| `--with-e2e` | (none) | E2E route: `cypress` (installs `@rancher/cypress` + runs `rancher-cypress init` post-install) or `playwright` (creates an isolated `tests/` sub-workspace à la [kubewarden-ui](https://github.com/rancher/kubewarden-ui/tree/main/tests)). |

**Version matrix (verified)**

| `@rancher/shell` | Rancher | Extensions API |
|---|---|---|
| `3.0.x`（含 `3.0.8`, `3.0.11`） | **2.13.x** | `>= 3.0.0 < 4.0.0` |
| `3.1.x` / `4.x` | unverified | derived, warns on stderr |

⚠️ Override `--shell-version` **only** if your target Rancher is not 2.13.x. The npm scaffold's own default may pull a newer/incompatible shell.

**The script does** (10 steps grouped in 3 layers):

- **① Scaffold** (steps 1–3): run `npm init @rancher/extension@latest`, pin `@rancher/shell` in top-level `package.json`, write `rancher.annotations` into `pkg/<name>/package.json`.
- **② Structure** (steps 4–6; `--minimal` skips 4–5): community-convention folders + `.gitkeep`, seed `types.ts` / READMEs / placeholder icon, patch `vue.config.js` with the `@rancher/shell 3.0.x` TS6059 rootDir workaround.
- **③ Engineering stack** (steps 7–10): always-on **lint scripts** (`lint` / `lint:fix`, no new devDeps — ESLint comes via `@rancher/shell`) → **Jest** by default (`--no-tests` skips: adds `jest` / `ts-jest` / `@vue/vue3-jest` / `jest-environment-jsdom`, writes `jest.config.ts` with `@shell/*` + `@<name>/*` mappers, `jest.setup.ts`, SVG transform, `coverage/` to `.gitignore`) → **E2E** if `--with-e2e` (Cypress deps + `rancher-cypress init` post-install, **or** an isolated `tests/` sub-workspace à la kubewarden) → `yarn install` (+ `yarn install` inside `tests/` for Playwright).

To undo any single step: delete the entries from `package.json` (lint / test scripts) or the corresponding files. Step 7 has no `--no-lint` flag — the cost is 2 lines of JSON.

**Common invocations**

```bash
# 典型场景：目标 Rancher 2.13.x（默认 shell 3.0.8 + Jest）
./scripts/init-extension.sh --name hello-top --display-name "Hello Top"

# Playwright E2E（kubewarden 风格，独立 tests/ 子工作空间）
./scripts/init-extension.sh --name hello-top --with-e2e playwright

# @rancher/cypress（复用 dashboard 主仓的 page objects）
./scripts/init-extension.sh --name hello-top --with-e2e cypress
```

## Workflow

### New extension

> 💡 **Prefer the scripted path** — the steps below are the manual/underlying reference. For a one-command bootstrap that also pins `@rancher/shell` and adds community-convention folders, use [`scripts/init-extension.sh`](scripts/init-extension.sh) (see the "Initializing a new extension" section above).

1. Read [references/getting-started.md](references/getting-started.md).
2. Ensure Node.js v24 (for `@latest` tag targeting Rancher v2.10+).
3. Scaffold a development app + first extension: `npm init @rancher/extension@latest <name>` (creates `./<name>/pkg/<name>/`).
4. `cd <name> && yarn install`.
5. Fill in `pkg/<name>/index.ts`, `product.ts`, `routing/extension-routing.ts`.
6. Run dev server: `API=<rancher-url> yarn dev` → open `https://127.0.0.1:8005` → extension is **auto-loaded with hot reload**. No manual Developer Load needed during development.
7. To test the **built** bundle: `yarn build-pkg <name>` → `yarn serve-pkgs` (port 4500) → in Rancher UI, **Extensions → Developer Load**, paste `http://127.0.0.1:4500/<name>-<version>/<name>-<version>.umd.min.js`.

### Add UI enhancement to existing views

1. Read [references/location-config.md](references/location-config.md) for LocationConfig.
2. For simple cases, use the inline minimal examples above.
3. For complex cases (all option keys, deprecated locations), open the specific API doc.
4. Add the enhancement in `index.ts` using `plugin.add*()`; do **not** rewrite unrelated parts of `index.ts`.

### Publish

1. Read [references/publishing.md](references/publishing.md).
2. Set `rancher.annotations` in `pkg/<name>/package.json`.
3. Use GitHub workflow (tagged release) or `yarn publish-pkgs`.

### Debug common errors

| Symptom | Likely cause |
|---------|--------------|
| `Cannot read properties of undefined (reading 'DSL')` in `product.ts` | `store` not injected into `$extension.DSL(store, name)`; check `init(plugin, args)` signature. |
| Route renders but side menu missing product entry | `product.ts` `init()` not called, or `inStore` mismatch between `product()` and `addRoutes()` meta. |
| Product silently missing after successful build (no error) | `plugin.addProduct` was called with ESM `import` / `export default`. See Key concepts › Extension entry pattern for the `require` + named `init` fix. |
| `TS2305: Module 'vue-router' has no exported member 'RouteConfig'` | vue-router v4 (Vue 3) renamed the type. Use `RouteRecordRaw` instead of `RouteConfig`. |
| Edit page saves with no network call | Missing `@finish="save"` on `CruResource`, or `CreateEditView` mixin not applied. |
| `addTab` component never renders | `LocationConfig` too strict (wrong `product`/`resource`), or `where` uses string literal instead of `TabLocation.*`. |
| Route renders as shell 404 (blank content) | Route `name` mismatch between the route definition and the `to.name` used in `product({...})` / side-menu registration. `meta.pkg` rarely helps — see Key concepts › Extension entry pattern. |
| Developer Load loads nothing / nav item missing | URL points to `:8005` (`yarn dev` server) instead of `:4500` (`yarn serve-pkgs`), or user is not admin. |
| Custom group label (e.g. "PaaS") reverts to raw API group name after installing another extension | The new extension used `plugin.DSL(store, 'explorer')` which registers `'explorer'` in its `productNames`. On any reload/uninstall, `removeProduct('explorer')` wipes all explorer state including `mapGroup` mappings. Fix: use `import { DSL } from '@shell/store/type-map'` instead. See [products.md § Hook-into-explorer](references/products.md#hook-into-explorer-pattern-parasitic-mode). |
| Extension disappears after browser reload | Loaded via Developer Load without `Persist extension by creating custom resource` checkbox. |
| l10n key shows as raw string | Key collides with core translation; scope under your extension namespace. |
| List page hangs for seconds on large clusters / memory balloons | `findAll` on a Steve type with thousands of instances. Probe `paginationEnabled` and branch to **`findPage`** under SSP-on. See [references/performance.md](references/performance.md). |
| Resource picker lags on large clusters | Using `LabeledSelect` — replace with **`ResourceLabeledSelect`** and configure `paginatedResourceSettings`. |
| Two views of the same resource type overwrite each other's data | Both paths write to the shared store cache via `findPage`. Switch one to **`PaginationWrapper`** to hold results outside the cache. |
| Jest fails with `Cannot find module '@shell/...'` | `moduleNameMapper` missing the shell alias. Add `'@shell/(.*)': '<rootDir>/node_modules/@rancher/shell/$1'` (and mirror every other alias in `vue.config.js`). See [references/testing.md](references/testing.md). |
| Cypress install fails with peer-dep conflict on `@rancher/cypress` | The package pins `cypress@11.1.0`. Either downgrade Cypress or switch to Playwright in a `tests/` sub-workspace. |
