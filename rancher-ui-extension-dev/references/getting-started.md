# Getting Started

> 💡 **Prefer the scripted path** — the manual steps below are the underlying reference. For a one-command bootstrap that also pins `@rancher/shell` and adds community-convention folders, see [`scripts/init-extension.sh`](../scripts/init-extension.sh) and the "Initializing a new extension" section of the SKILL.

Official source: https://extensions.rancher.io/extensions/next/extensions-getting-started

This guide walks through setting up the extension development environment and creating a first extension.

## Two-concept mental model

| Concept | What it is | Where it lives |
|---|---|---|
| **Development app** | A full Rancher UI shell used to host and hot-reload extensions during development. Wraps `@rancher/shell` via npm. | A folder generated by `npm init @rancher/extension@latest <name>`. |
| **Extension / package** | The actual extension source code (one or many per development app). | `./pkg/<extension-name>/` inside the development app. |

You can host **multiple extensions in one development app**; they will all hot-reload together during `yarn dev`.

## Prerequisites

> Extensions development is supported only on macOS and Linux. Windows is not supported.

- **Node.js `v24`** (tested with `v24.14.0`) for the `latest` tag.
- Yarn global: `npm install -g yarn`.
- A running Rancher instance — see the [Rancher installation docs](https://ranchermanager.docs.rancher.com/getting-started/installation-and-upgrade).

Node version is bound to the `@rancher/extension` tag. All extensions inside the same development app **must share the same tag**, otherwise you'll hit compatibility errors.

## Scaffolding

Rancher publishes a single npm package `@rancher/extension` (replacing the legacy `@rancher/app` + `@rancher/pkg`).

**Version matching rule**:

| Target Rancher version | Tag to use |
|---|---|
| `v2.10` or later | `latest` |

### Common invocations

```sh
# Create a development app "my-app" AND an extension package "my-app" (both share the same name)
npm init @rancher/extension@latest my-app
cd my-app && yarn install
```

```sh
# Add another extension package inside an existing development app
# (run this from the development app root)
npm init @rancher/extension@latest another-extension
```

```sh
# Development app and extension with different names
npm init @rancher/extension@latest new-extension -- --app-name my-app
```

```sh
# Just the development app (no extension package yet)
npm init @rancher/extension@latest my-app -- --skeleton-only
```

### CLI options

| Option | Effect |
|---|---|
| `--update` / `-u` | Update all deps in the extension to match `@rancher/shell` version for the chosen tag. |
| `--app-name <name>` / `-a` | Development app name (defaults to extension name). |
| `--skeleton-only` / `-s` | Create only the development app; skip the extension package. |
| `-l` | Include `.gitlab-ci.yml` for GitLab CI integration. |
| `-w` | **Skip** GitHub workflow files (`build-extension-catalog.yml`, `build-extension-charts.yml`). By default these are included. |
| `-t` | **Skip** template folders inside the extension package. By default these are included. |

Options that use double values (`--app-name`, `--skeleton-only`) require the `--` separator when passed to `npm init`.

## Running the development app

```sh
yarn install
API=<Rancher Backend URL> yarn dev
```

- Open `https://127.0.0.1:8005` → you get a full Rancher UI referencing your Rancher backend.
- Your extension inside `./pkg/<name>/` is **auto-loaded with hot reload**. No manual "Developer Load" needed at this stage.
- Admin login is required to test extensions.

## Basic extension example (hello-world)

By the end you'll have a new sidebar entry in Rancher that opens a custom page. Assumes the extension is called `my-app`.

### 1. `./pkg/my-app/index.ts`

```ts
import { importTypes } from '@rancher/auto-import';
import { IPlugin } from '@shell/core/types';
import extensionRouting from './routing/extension-routing';

// Init the package
export default function (plugin: IPlugin): void {
  // Auto-import model, detail, edit from conventional folders
  importTypes(plugin);

  // Provide extension metadata from package.json (name, version, description)
  plugin.metadata = require('./package.json');

  // Load a product
  plugin.addProduct(require('./product'));

  // Add Vue Routes
  plugin.addRoutes(extensionRouting);
}
```

What each call does:

| Call | Purpose |
|---|---|
| `importTypes(plugin)` | Auto-registers components in `./models/`, `./detail/`, `./edit/`. Safe to omit if you have no custom resource views. |
| `plugin.metadata` | Exposes name/version/description from `package.json` to the Extensions marketplace. Without this, installed extensions have no display name. |
| `plugin.addProduct` | Loads `./product.ts` and registers the product's navigation entry. |
| `plugin.addRoutes` | Registers Vue Router routes so Rancher knows which component to render at each URL. |

### 2. `./pkg/my-app/product.ts`

```ts
import { IPlugin } from '@shell/core/types';

export function init($extension: IPlugin, store: any): void {
  const YOUR_PRODUCT_NAME = 'myProductName';
  const BLANK_CLUSTER = '_';

  const { product } = $extension.DSL(store, YOUR_PRODUCT_NAME);

  product({
    icon:    'gear',
    inStore: 'management',
    weight:  100,
    to: {
      name: `${ YOUR_PRODUCT_NAME }-c-cluster`,
      path: `/${ YOUR_PRODUCT_NAME }/c/:cluster/dashboard`,
      params: {
        product: YOUR_PRODUCT_NAME,
        cluster: BLANK_CLUSTER,
        pkg:     YOUR_PRODUCT_NAME,
      },
    },
  });
}
```

> **`BLANK_CLUSTER = '_'`**: Rancher convention meaning "this product does not operate in the context of a specific downstream cluster". Global products (Fleet, Cluster Management) use `'_'`. For a cluster-scoped product use `inStore: 'cluster'` instead — see [concepts.md](concepts.md).

> **`meta.pkg` / `params.pkg`** *(optional)*: The extension package name. Rancher uses it to know which extension "owns" the route for lazy-loading. **Most official examples (top-level-product, clock, shell-api-demo, homepage) omit `pkg` and render fine.** Add it only if a complex routing setup renders the shell 404 despite the route `name` matching — or when you need precise bundle lazy-load boundaries.

### 3. `./pkg/my-app/routing/extension-routing.ts`

```ts
import { RouteRecordRaw } from 'vue-router';
import Dashboard from '../pages/index.vue';

const BLANK_CLUSTER = '_';
const YOUR_PRODUCT_NAME = 'myProductName';

const routes: RouteRecordRaw[] = [
  {
    name:      `${ YOUR_PRODUCT_NAME }-c-cluster`,
    path:      `/${ YOUR_PRODUCT_NAME }/c/:cluster`,
    component: Dashboard,
    meta:      {
      product: YOUR_PRODUCT_NAME,
      cluster: BLANK_CLUSTER,
      pkg:     YOUR_PRODUCT_NAME,
    },
  },
];

export default routes;
```

> Don't forget to create the referenced `./pages/index.vue`.

Save all three files while `yarn dev` is running → hot reload → the new nav item appears in Rancher's main sidebar.

## Building for distribution

```sh
yarn build-pkg my-app
```

Output lands in `./dist-pkg/`. This produces a Vue library (UMD bundle) that can be dynamically loaded into any Rancher instance.

## Loading a built extension (Developer Load)

Two URLs, two situations — do not confuse them:

| Scenario | Command | URL to paste into "Developer Load" |
|---|---|---|
| **Live development** | `yarn dev` (port `8005`) | *No manual load needed — auto-loaded with hot reload.* |
| **Test the built bundle** | `yarn serve-pkgs` (port `4500`) | `http://127.0.0.1:4500/<name>-<version>/<name>-<version>.umd.min.js` |

Steps for the built-bundle scenario:

1. **Enable Developer Features** in Rancher UI:
   User avatar (top-right) → **Preferences** → **Advanced Features** → check **Enable Extension developer features**.

2. **Serve the built package**:
   ```sh
   yarn serve-pkgs
   ```
   Example output:
   ```
   Serving catalog on http://127.0.0.1:4500

   Serving packages:

     my-app-0.1.0 available at: http://127.0.0.1:4500/my-app-0.1.0/my-app-0.1.0.umd.min.js
   ```

3. **Load in the UI**:
   Slide-in menu (hamburger, top-left) → **Extensions** → three-dot menu → **Developer load** → paste the URL above → **Load**.

4. **Persist across page reload**:
   Check **Persist extension by creating custom resource** in the Developer Load dialog. Otherwise the extension unloads on refresh.

### Testing dynamic load while `yarn dev` is running

To simulate "extension is not baked into the app" behavior, exclude it from the dev bundle by editing `vue.config.js` at the development app root:

```js
const config = require('@rancher/shell/vue.config');

module.exports = config(__dirname, {
  excludes: ['my-app'],
});
```

Then restart `API=<url> yarn dev`. The nav item disappears; use Developer Load to load the built bundle instead.

## Troubleshooting quick reference

| Symptom | Likely cause |
|---|---|
| `yarn install` fails with node engine mismatch | Wrong Node version. Match Node to the tag (v24 for `latest`). |
| Multiple extensions in same app fail to build | Extensions use different `@rancher/extension` tags. Align all extensions **and** the dev app to one tag. |
| Nav item does not appear after Developer Load | Not logged in as admin, or extension URL points to `8005` (dev server) instead of `4500` (serve-pkgs). |
| Route renders shell 404 | First verify route `name` matches `to.name` in `product({...})`. In complex routing setups, adding `meta.pkg: <extension-name>` may help. Most official examples omit `pkg` entirely. |
| Extension disappears after refreshing the browser | Developer Load without "Persist extension by creating custom resource" enabled. |
| `yarn dev` fails with `TS6059 ... is not under 'rootDir' '.../node_modules/@rancher/shell'` | Known bug in `@rancher/shell 3.0.x` — ts-loader inherits `rootDir` from the shell's own `tsconfig.json`. Fixed upstream in [rancher/dashboard#17119](https://github.com/rancher/dashboard/pull/17119) (milestone v2.15.0), not yet released. Workaround: patch the project's `vue.config.js` to override ts-loader `compilerOptions.rootDir` to `__dirname` (the [`init-extension.sh`](../scripts/init-extension.sh) script generates this workaround automatically; see the script's `patch vue.config.js` step for the exact snippet). After patching, clear `node_modules/.cache` and retry. |
